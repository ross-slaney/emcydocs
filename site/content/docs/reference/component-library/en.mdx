## Accordions

<Accordion>
  <AccordionItem title="What belongs in an accordion?" defaultOpen>
    Put FAQs, optional setup branches, migration caveats, or product-specific notes here. Keep the title short and action-oriented.
  </AccordionItem>
  <AccordionItem title="Does it still work with normal markdown?">
    Yes. You can mix fenced code blocks, headings, lists, tables, and JSX components in the same `.mdx` file.
  </AccordionItem>
  <AccordionItem title="Why not depend on another UI package?">
    EmcyDocs keeps everything in one library on purpose. The components are small, docs-focused, and styled by the same shared stylesheet.
  </AccordionItem>
</Accordion>

    <Steps>
      <Step title="Import the package once">

      </Step>
      <Step title="Use the shared layouts">
        Start with `DocsLayout`, `NotebookLayout`, or `MinimalLayout` and then add your own search or locale switching.
      </Step>
      <Step title="Reach for components in MDX">
        Use banners, cards, accordions, files, tabs, and steps where a plain markdown block would feel flat.
      </Step>
    </Steps>
  </TabsContent>

## Code blocks

<CodeBlock title="layout.tsx" language="tsx">{`import { DocsLayout, ThemeSwitcher } from "@emcy/docs";
import "@emcy/docs/styles.css";

export default function Layout({ children }) {
  return (
    <DocsLayout
      navigation={docsSource.getNavigation()}
      theme={{
        preset: "neutral",
        mode: "light",
        layoutWidth: "1500px",
        contentWidth: "48rem",
        radius: "xl",
      }}
      themeSwitcher={<ThemeSwitcher />}
    >
      {children}
    </DocsLayout>
  );
}`}</CodeBlock>

The goal is straightforward: one package, one visual language, and enough primitives to make docs pages look intentional without creating another package boundary.
